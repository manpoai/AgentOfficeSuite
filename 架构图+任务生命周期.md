## 一、整体架构图

```mermaid
flowchart TD
    subgraph Human["👤 Human"]
        MM_H["Mattermost\n@mention / DM"]
        OL_H["Outline\n评论 @mention"]
        PL_H["Plane\n分配任务 / 评论"]
    end

    subgraph GW["🔀 ASuite Gateway（协议翻译层）"]
        direction TB
        REG["注册 + Provisioning\nPOST /api/auth/register\n→ 自动创建文档集、MM Bot"]
        CTX["Agent Context Injection\n补全缺省参数\n（collection_id、发件人等）"]
        MM_WS["MM WebSocket 监听"]
        OL_WH["Outline Webhook"]
        PL_POLL["Plane 轮询（15s）"]
        EVT["事件标准化\nmessage.mentioned / task.assigned\ncomment.mentioned / task.commented"]
        API["统一操作 API\n/api/messages  /api/docs\n/api/tasks  /api/comments"]
        SSE["SSE 推送流"]
        DB[("events DB")]
    end

    subgraph Adapter["🔌 Adapter（接入层，per-agent）"]
        SSE_C["SSE 客户端"]
        FMT["格式转换 → 自然语言消息"]

    end

    subgraph Agent["🤖 Agent（执行层）"]
        QUEUE["C4 消息队列"]
        CLAUDE["Claude Code"]
        REPLY["c4-send.js 回复路由"]
    end

    MM_H -->|"@mention / DM"| MM_WS
    OL_H -->|"webhook"| OL_WH
    PL_H -->|"assign / comment"| PL_POLL

    MM_WS --> EVT
    OL_WH --> EVT
    PL_POLL --> EVT

    EVT --> DB --> SSE --> SSE_C
    SSE_C --> FMT --> C4 --> QUEUE --> CLAUDE

    CLAUDE -->|"POST /api/tasks 等"| CTX
    CTX -->|"补全 collection_id 等"| API
    REPLY --> CTX

    API -->|"→ Mattermost"| MM_H
    API -->|"→ Outline"| OL_H
    API -->|"→ Plane"| PL_H

    REG -.->|"入驻时创建资源\n写入 workspace 配置"| CTX
```

**各层职责：**

| 层 | 职责 | 不做什么 |
|---|---|---|
| Gateway 注册+Provisioning | 身份创建 + 工作空间初始化（文档集、Bot） | 不执行业务逻辑 |
| Gateway Context Injection | 用 Agent 身份补全缺省参数 | 不改变 Agent 意图 |
| Gateway 事件分发 | 协议翻译、SSE 推送、Catchup | 不执行业务逻辑 |
| Adapter | SSE 监听、格式转换、C4 注入 | 不调用 API、不做业务判断 |
| Agent | 理解任务、执行工作、更新状态 | 不直接对接 MM/Outline/Plane |

## 四、任务生命周期状态机

```mermaid
stateDiagram-v2
    [*] --> todo : 任务创建

    todo --> in_progress : Agent 收到任务\n立即 ACK
    todo --> cancelled : Human 取消

    in_progress --> done : Agent 完成工作\n主动更新状态
    in_progress --> cancelled : Human 取消

    done --> [*]
    cancelled --> [*]
```

**核心原则：Gateway 不自动推进任何状态。所有状态变更由 Agent 主动调用 `PATCH /api/tasks/:id/status` 完成。**

| 状态 | 含义 | 由谁设置 |
|---|---|---|
| `todo` | 已创建，等待执行 | 创建时默认 |
| `in_progress` | 执行中 | Agent 收到任务后立即调用 API |
| `done` | 已完成 | Agent 完成后调用 API |
| `cancelled` | 已取消 | Human 或 Agent 均可 |

## 五、Agent-to-Agent 委派时序

```mermaid
sequenceDiagram
    participant H as Human
    participant A as Agent A (Zylos)
    participant GW as Gateway
    participant B as Agent B (Digger)

    H->>A: @mention 请求
    A->>GW: POST /api/tasks\n{ assignee: digger, context: "..." }
    GW-->>B: task.assigned 事件\n（含 delegation_context）
    B->>GW: PATCH status → in_progress
    Note over B: 执行工作
    B->>GW: POST comment @zylos + PATCH status → done
    GW-->>A: task.commented 事件
    A->>H: 汇报结果
```

## 七、v1.1 待完善方向

1. **全渠道任务可视化**：来自 IM 和 Outline 的触发，自动在 Plane 创建任务记录，所有工作都在 Plane 可见
2. **任务排队语义**：Backlog（已知未开始）vs Todo（就绪待取），Agent 按优先级顺序取任务
3. **超时检测**：`in_progress` 超过阈值未变化，Gateway 推 `task.timeout` 事件，通知 Human 介入
4. **资源隔离**：Agent 默认只读写自己 Collection，跨 Collection 需显式声明
5. **署名注入**：Agent 创建的内容自动带来源元数据
